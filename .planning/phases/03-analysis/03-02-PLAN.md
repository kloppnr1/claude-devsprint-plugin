Create `~/.claude/commands/gsd/azdo-analyze.md` following the established skill file pattern from azdo-sprint.md.

**Frontmatter:**
```yaml
name: gsd:azdo-analyze
description: Analyze sprint stories and bootstrap GSD projects in target repos
argument-hint: ""
allowed-tools:
  - Read
  - Write
  - Bash
  - AskUserQuestion
```

**Process (numbered steps the skill executor follows):**

**Step 1 — Check prerequisites:**
- Verify `~/.claude/get-shit-done/bin/azdo-tools.cjs` exists. If not, tell user and stop.
- Run `load-config --cwd $CWD`. If exit 1, tell user to run `/gsd:azdo-setup`. Stop.

**Step 2 — Fetch sprint metadata:**
- Run `get-sprint --cwd $CWD`. Parse JSON. Extract sprint name for display.

**Step 3 — Fetch assigned stories:**
- Run `get-sprint-items --me --cwd $CWD`. Parse JSON array.
- Filter to top-level stories only: items where `type === "User Story"` AND (`parentId === null` OR parentId is not in the items list). This is the story-level-only decision.
- If a user passes a task ID argument, find the item in the list and use its `parentId` to roll up to the parent story (story-level only per locked decision).
- Collect child tasks for each story: items where `parentId === storyId`.

**Step 4 — Resolve branch links:**
- For each top-level story, run `get-branch-links --id {storyId} --cwd $CWD`. Parse JSON array.
- If multiple branch links for one story, use the first one (Claude's discretion per RESEARCH.md).
- Group stories by target repo (by `repositoryName`).
- Track stories with no branch links (empty array) separately.

**Step 5 — Show multi-repo summary and confirm:**
Display summary in this format:
```
=== Analysis: {sprintName} ===

You have {N} stories across {M} repos:

Repo: {repoName} ({remoteUrl})
  [US] #{id} -- {title} ({state})
  [US] #{id} -- {title} ({state})

Repo: {repoName} ({remoteUrl})
  [US] #{id} -- {title} ({state})

{K} story/stories have no branch link and will be skipped:
  [US] #{id} -- {title} ({state})

Proceed to generate GSD projects for {M} repos? (yes/no)
```
Use `AskUserQuestion` tool for confirmation. If "no", stop.

**Step 6 — For each target repo, resolve local path:**
- Extract repo name from `remoteUrl`: last segment of the URL path.
- Check if repo exists at `{parent of $CWD}/{repoName}` (same parent directory as PlanningMe repo, per observed layout at `C:/Users/sen.makj/source/repos/`).
- If found (`.git` directory exists there), use that path.
- If NOT found, use `AskUserQuestion` to ask: "Repo {repoName} not found locally at {expectedPath}. Options: (1) Clone it there, (2) Provide a different path, (3) Skip this repo".
  - If clone: run `git clone "{remoteUrl}" "{expectedPath}"` via Bash. If clone fails, warn and skip.
  - If different path: use the provided path.
  - If skip: exclude this repo from generation.

**Step 7 — Check for existing .planning/ in target repo:**
- If `{repoPath}/.planning/PROJECT.md` already exists, use `AskUserQuestion`: "Repo {repoName} already has a GSD project at {repoPath}/.planning/. Overwrite? (yes/no)". If no, skip this repo.

**Step 8 — Generate PROJECT.md for each target repo:**
Use the `Write` tool to create `{repoPath}/.planning/PROJECT.md`. Ensure `.planning/` directory exists first (create via Bash `mkdir -p` if needed).

Map Azure DevOps fields to GSD PROJECT.md sections per the mapping in RESEARCH.md Pattern 4:
- `# {repo.name}` as project header
- `## What This Is` from story description (first 2-3 sentences, cleaned). Include "This work is tracked as Azure DevOps story #{id}: "{title}"."
- `## Core Value` derived from story description (the single most important thing)
- `## Requirements > Active` from acceptanceCriteria (each criterion as a checkbox) PLUS child task titles as additional checkboxes
- `## Context` with Azure DevOps metadata: story ID, sprint name, branch name, state
- `## Constraints` with tech stack inferred from repo name/description

Read the GSD template at `~/.claude/get-shit-done/templates/project.md` for exact section structure to follow.

**Step 9 — Generate ROADMAP.md for each target repo:**
Use the `Write` tool to create `{repoPath}/.planning/ROADMAP.md`.

Map Azure DevOps fields to GSD ROADMAP.md:
- `# Roadmap: {story title}`
- 1-2 phases (keep it simple — single story = single phase is fine)
- Phase success criteria from acceptance criteria
- Phase plans from child task groupings

Read the GSD template at `~/.claude/get-shit-done/templates/roadmap.md` for exact section structure.

Also create `{repoPath}/.planning/REQUIREMENTS.md` with requirement IDs derived from acceptance criteria and child tasks.

**Step 10 — Present for approval per repo:**
For each target repo, show the generated PROJECT.md and ROADMAP.md content to the user. Use `AskUserQuestion`: "Review the generated GSD project for {repoName}. Approve, request changes, or skip? (approve/changes/skip)".
- If "approve": keep the written files, move to next repo.
- If "changes": ask what to change, regenerate the files with the feedback, re-present.
- If "skip": delete the generated files from that repo.

**Step 11 — Final summary:**
```
=== Analysis Complete ===

GSD projects bootstrapped:
  {repoName}: {repoPath}/.planning/ (approved)
  {repoName}: {repoPath}/.planning/ (approved)

Skipped:
  {repoName}: no branch link
  {repoName}: user skipped

Next: cd {repoPath} && /gsd:plan-phase
```

**Success criteria section in the skill:**
- All assigned stories resolved via branch links
- PROJECT.md and ROADMAP.md generated in each approved target repo
- User approved each generated project before finalization
- No HTML artifacts in generated files
